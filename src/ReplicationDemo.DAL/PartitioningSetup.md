# Partitioning Setup — ReplicationDemo.DAL

This document describes how the `JobExecutions` table is physically partitioned in the SQL Server database, how EF Core interacts with the partitioned table, and how to perform ongoing maintenance.

---

## Strategy Summary

| Attribute | Value |
|---|---|
| **Table** | `dbo.JobExecutions` |
| **Strategy** | Range partitioning (RANGE RIGHT) |
| **Partition key** | `StartedAt` (DATETIME2) |
| **Granularity** | One partition per calendar month |
| **Partition function** | `PF_JobExecutions_ByMonth` |
| **Partition scheme** | `PS_JobExecutions_ByMonth` |
| **Filegroup** | `[PRIMARY]` (all partitions) |

See [Partitioning-Sharding-Strategy.md](../../Partitioning-Sharding-Strategy.md) for the full decision rationale.

---

## Partition Layout

```
Partition │ Range                           │ Contents
──────────┼─────────────────────────────────┼──────────────────────
    1     │ StartedAt <  '2026-01-01'       │ catch-all (pre-history)
    2     │ '2026-01-01' ≤ x < '2026-02-01' │ January  2026
    3     │ '2026-02-01' ≤ x < '2026-03-01' │ February 2026
    4     │ '2026-03-01' ≤ x < '2026-04-01' │ March    2026
    5     │ '2026-04-01' ≤ x < '2026-05-01' │ April    2026
    6     │ '2026-05-01' ≤ x < '2026-06-01' │ May      2026
    7     │ '2026-06-01' ≤ x < '2026-07-01' │ June     2026  ← current
    8     │ StartedAt ≥  '2026-07-01'       │ future buffer
```

RANGE RIGHT means each boundary value belongs to the **right** (higher) partition. New rows with the current month's `StartedAt` always land in the last non-future partition.

## Index Design

### Clustered Primary Key — `PK_JobExecutions` on `(StartedAt, Id)`

```sql
CONSTRAINT PK_JobExecutions PRIMARY KEY CLUSTERED (StartedAt, Id)
    ON PS_JobExecutions_ByMonth (StartedAt)
```

- SQL Server requires the partition key to be part of every unique/primary index on a partitioned table.
- `StartedAt` leads the key so sequential time-range scans stay within a single partition (no cross-partition I/O).

### Unique Non-Clustered — `UX_JobExecutions_Id` on `(Id)`

```sql
CREATE UNIQUE NONCLUSTERED INDEX UX_JobExecutions_Id
    ON dbo.JobExecutions (Id);
```

- Non-aligned (spans all partitions).
- Lets EF Core resolve `GetByIdAsync(Guid id)` → `WHERE Id = @id` without supplying `StartedAt`.

### Non-Clustered — `IX_JobExecutions_JobId_StartedAt` on `(JobId, StartedAt DESC)`

```sql
CREATE NONCLUSTERED INDEX IX_JobExecutions_JobId_StartedAt
    ON dbo.JobExecutions (JobId, StartedAt DESC)
    ON PS_JobExecutions_ByMonth (StartedAt);   -- partition-aligned
```

- Supports `GetExecutionsByJobIdAsync`: `WHERE JobId = @id ORDER BY StartedAt DESC`.
- Partition elimination applies automatically when a `StartedAt` range predicate is added to the query.

## EF Core Compatibility

EF Core is configured with `HasKey(e => e.Id)` in `JobExecutionConfiguration`. This is a **logical entity key** used for change tracking; it does not control the physical clustered index structure.

Because the database schema is managed entirely by SQL initialization scripts (no EF Core migrations), there is no conflict between the composite clustered PK `(StartedAt, Id)` in SQL Server and the single-column entity key `Id` used by EF Core.

All CRUD operations generated by EF Core are compatible:

| Operation | Generated SQL | Works with partition? |
|---|---|---|
| Insert | `INSERT INTO JobExecutions (JobId, StartedAt, …) VALUES (…)` | ✅ — row lands in correct partition automatically |
| Select by Id | `WHERE Id = @id` | ✅ — uses `UX_JobExecutions_Id` |
| Select by JobId | `WHERE JobId = @id ORDER BY StartedAt DESC` | ✅ — uses `IX_JobExecutions_JobId_StartedAt` |
| Delete by Id | `WHERE Id = @id` | ✅ — uses `UX_JobExecutions_Id` to locate row |

## Monthly Maintenance

### Add the next month's boundary (run before the month starts)

```sql
-- Step 1: designate the next filegroup for the new partition
ALTER PARTITION SCHEME PS_JobExecutions_ByMonth
    NEXT USED [PRIMARY];

-- Step 2: split the "future buffer" partition at the new boundary
ALTER PARTITION FUNCTION PF_JobExecutions_ByMonth()
    SPLIT RANGE ('2026-08-01');   -- replace with the actual next month
```

### Archive and drop an old partition (sliding-window pattern)

```sql
-- 1. Create a staging table with the same schema (no data)
CREATE TABLE dbo.JobExecutions_Archive_2026_01 (
    Id         UNIQUEIDENTIFIER NOT NULL,
    JobId      UNIQUEIDENTIFIER NOT NULL,
    StartedAt  DATETIME2        NOT NULL,
    FinishedAt DATETIME2        NULL,
    Result     NVARCHAR(50)     NOT NULL,
    CreatedAt  DATETIME2        NOT NULL,
    CONSTRAINT PK_JobExecutions_Archive_2026_01
        PRIMARY KEY CLUSTERED (StartedAt, Id)
) ON [PRIMARY];

-- 2. Switch partition 2 (January 2026) out to the archive table (O(1) metadata operation)
ALTER TABLE dbo.JobExecutions
    SWITCH PARTITION 2
    TO dbo.JobExecutions_Archive_2026_01;

-- 3. Merge the now-empty partition boundary to compact the function
ALTER PARTITION FUNCTION PF_JobExecutions_ByMonth()
    MERGE RANGE ('2026-01-01');

-- 4. Optionally drop or export the archive table
-- DROP TABLE dbo.JobExecutions_Archive_2026_01;
```

## Replication Note

The partition function and scheme are created on both the **primary** and **replica** servers by `01-schema.sql`. Transactional replication propagates row-level DML (INSERT/UPDATE/DELETE), so new execution records written to the primary are automatically delivered to the corresponding partition on the replica via the Log Reader and Distribution agents.
